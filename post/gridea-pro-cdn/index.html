从手动上传到一键部署：Gridea Pro CDN 图床自动上传功能的设计与实现

2026年03月12日 · 10 分钟阅读
Gridea Pro CDN 图床 GitHub API 技术实现 静态博客

你刚写完一篇图文并茂的技术博客，插入了 12 张截图，点击「部署」，静静等待站点上线。打开页面一看——所有图片全是裂开的。

你猛然想起：图片还躺在本地，忘了手动上传到 CDN 仓库。

于是你打开 GitHub，一个一个拖文件进去，等 commit 完成，再回来重新部署。下次写文章，同样的流程再来一遍。

这件事不应该是手动的。

这篇文章记录了我们如何在 Gridea Pro 中实现 CDN 图床的自动上传功能——从原理分析、方案设计到最终落地，完整复盘整个过程。

一、先搞清楚：CDN 图床到底在做什么

在聊实现之前，先厘清几个概念。

静态博客的图片困境

Gridea Pro 是一个本地优先的静态博客客户端。你在本地写 Markdown，插入图片，图片存储在本地的 post-images/ 目录里。本地预览时一切正常，因为图片就在你的硬盘上。

但部署到线上就不一样了。静态站点托管平台（GitHub Pages、Vercel、Netlify）托管的是 HTML/CSS/JS，图片如果也塞进去，仓库会迅速膨胀，clone 变慢，构建超时。

CDN 图床就是解决方案：把图片存到一个专门的 GitHub 仓库，通过 jsDelivr 这样的 CDN 服务加速访问。渲染 HTML 时，把本地路径 /post-images/cover.png 替换为 https://cdn.jsdelivr.net/gh/user/repo@main/post-images/cover.png。

之前的方案：只做 URL 重写

Gridea Pro 之前已经有 CDN 功能，但只做了「最后一公里」——HTML 渲染时的 URL 重写。它假设你已经手动把图片上传到了 CDN 仓库。

这带来三个问题：

  1. 每次部署前要手动上传图片，容易遗漏
  2. 新增图片需要记住哪些还没上传，心智负担重
  3. 多设备写作时更麻烦，每台设备都要手动同步

我们要做的，就是补上「上传」这一环。

二、方案设计：看似简单，实则有坑

核心需求拆解

  1. 部署时自动上传：渲染完 HTML 后、推送到托管平台前，自动把媒体文件上传到 CDN 仓库
  2. 可配置的保存路径：支持按年月归档，避免文件堆积
  3. GitHub Token 配置：通过 API 上传需要认证
  4. 测试上传验证：配置完能立即验证是否生效
  5. 本地预览不受影响：预览时继续使用本地路径

保存路径的设计

参考 uPic 等成熟图床工具，我们设计了路径模板变量系统：

变量 含义 示例
{year} 当前年份 2026
{month} 当前月份 03
{day} 当前日期 12
{filename} 不含扩展名的文件名 cover
{.suffix} 带点的扩展名 .png
{suffix} 不带点的扩展名 png
{random} 8 位随机字符串 a1b2c3d4
{since_second} Unix 时间戳（秒） 1773504000

预设了 4 种常用模式：

{year}/{month}/{filename}{.suffix}           → 2026/03/cover.png
{year}/{month}/{day}/{filename}{.suffix}     → 2026/03/12/cover.png
blog/images/{year}/{month}/{filename}{.suffix} → blog/images/2026/03/cover.png
{year}/{filename}-{random}{.suffix}          → 2026/cover-a1b2c3d4.png

也支持完全自定义。

一个差点踩进去的坑：批量上传的路径问题

最初的设计是：部署时也用 savePath 模板来决定每个文件的远程路径。但仔细一想，这有严重问题——

假设 savePath 是 {year}/{month}/{filename}{.suffix}，1 月份写的文章里的图片上传后路径是 2026/01/post-images/cover.png，CDN base URL 变成 https://cdn.jsdelivr.net/gh/user/repo@main/2026/01。到了 3 月再次部署，路径变成 2026/03，1 月份的旧图片 URL 全部失效。

修正方案：批量部署上传时，保持原始目录结构不变（post-images/xx、images/xx、media/xx），直接映射到 CDN 仓库根目录。buildCdnBase() 的逻辑不动。savePath 模板仅用于测试上传和未来的编辑器内单图上传场景。

这样，无论何时部署，HTML 中的 /post-images/cover.png 始终重写为 https://cdn.jsdelivr.net/gh/user/repo@main/post-images/cover.png，路径稳定不变。

三、技术实现

整体架构

用户点击部署
  → 渲染静态站点 (RenderAll)
  → CDN 上传媒体文件 (UploadMediaForDeploy)  ← 新增
  → 推送到托管平台 (Deploy)

这个流程嵌入在 DeployService.DeployToRemote() 中，渲染之后、平台部署之前。

GitHub Contents API 上传

GitHub 提供了 Contents API 来读写仓库中的单个文件，非常适合我们的场景：

检查文件是否存在：

GET /repos/{owner}/{repo}/contents/{path}?ref={branch}

创建或更新文件：

PUT /repos/{owner}/{repo}/contents/{path}
{
  "message": "Upload post-images/cover.png",
  "content": "<base64 encoded>",
  "branch": "main",
  "sha": "<existing sha if updating>"
}

SHA 去重：避免重复上传

每次部署都要扫描所有媒体文件，但大多数文件上次已经上传过了。如何避免重复劳动？

Git 对每个文件（blob）有一个 SHA1 指纹，计算方式是：

SHA1("blob {size}\0{content}")

我们在本地用同样的算法计算文件 SHA，与 GitHub API 返回的 SHA 比对。相同则跳过，不同才上传。这样即使有 100 张图片，如果只新增了 2 张，实际只会发起 2 次上传请求。

核心代码：

func gitBlobSHA(data []byte) string {
    header := fmt.Sprintf("blob %d\x00", len(data))
    h := sha1.New()
    h.Write([]byte(header))
    h.Write(data)
    return hex.EncodeToString(h.Sum(nil))
}

并发控制

扫描出的文件可能很多，串行上传太慢。我们用 Go 的 errgroup + channel 实现了 5 并发的上传管线：

g, gCtx := errgroup.WithContext(ctx)
sem := make(chan struct{}, 5) // 限制 5 并发

for _, file := range filesToUpload {
    f := file
    g.Go(func() error {
        sem <- struct{}{}        // 获取信号量
        defer func() { <-sem }() // 释放信号量
        return s.uploadToGitHub(gCtx, setting, f.localPath, f.remotePath)
    })
}

g.Wait()

为什么限制 5 而不是更多？GitHub API 有速率限制（认证用户 5000 次/小时），并发太高容易触发 secondary rate limit（短时间内请求过密）。5 是一个安全又高效的平衡点。

路径模板解析

解析逻辑很直观，用 strings.NewReplacer 做一次性替换：

func ResolveSavePath(template, filename string) string {
    now := time.Now()
    ext := filepath.Ext(filename)
    nameOnly := strings.TrimSuffix(filename, ext)

    replacer := strings.NewReplacer(
        "{year}", now.Format("2006"),
        "{month}", now.Format("01"),
        "{day}", now.Format("02"),
        "{filename}", nameOnly,
        "{.suffix}", ext,
        "{suffix}", strings.TrimPrefix(ext, "."),
        "{random}", randomString(8),
        // ... 更多变量
    )

    return replacer.Replace(template)
}

每个文件独立解析，{random} 每次调用生成不同值，保证文件名不冲突。

四、前端交互设计

Token 输入

GitHub Token 是敏感信息，默认用密码框遮盖，右侧有眼睛图标切换可见性。这是个小细节，但体验差距很大——谁也不想在演示屏幕上暴露自己的 Token。

保存路径选择

用下拉框提供 4 个预设方案，底部还有「自定义」选项。选择自定义后展开输入框，下方提示支持的变量列表。这样既照顾了只想快速配置的用户，也给高级用户留了空间。

测试上传

配置完点「测试上传」，会上传一个小的测试文件到 CDN 仓库，成功后 toast 显示完整的 CDN 访问 URL。用户可以直接点击验证是否能访问。这比「保存后祈祷部署时能用」的体验好太多了。

测试按钮只在满足条件时显示（CDN 已启用 + jsDelivr 模式 + Token 已填写），避免无效点击。

五、完整数据流回顾

1. 用户在 CDN 设置页配置：
   - 启用 CDN ✓
   - Provider: jsDelivr
   - GitHub User/Repo/Branch
   - GitHub Token
   - 保存路径模板

2. 用户写文章，插入图片 → 图片保存到本地 post-images/

3. 本地预览 → 直接读本地文件，图片正常显示

4. 用户点击部署：
   ① RenderAll() — 生成 HTML，CDN URL 重写生效
   ② UploadMediaForDeploy() — 扫描 post-images/、images/、media/
      → 逐个计算 SHA，与 GitHub 比对
      → 新文件或内容变更的文件上传到 CDN 仓库
      → 内容相同的文件跳过
   ③ Deploy() — 推送 HTML 到托管平台

5. 用户访问线上站点 → 图片从 jsDelivr CDN 加载 ✓

六、一些取舍和思考

为什么用 GitHub Contents API 而不是 Git Push

Contents API 的好处是简单——不需要在用户机器上 clone CDN 仓库，不需要处理 git 冲突，每个文件独立上传。坏处是单次请求有 100MB 限制（但博客图片很少超过这个），以及大量文件时的速率问题。

对于博客图床这个场景，Contents API 是更合适的选择。如果未来需要支持更大规模的文件（比如视频），可以考虑切换到 Git Data API（blob + tree + commit），一次 commit 上传所有文件。

为什么单文件失败不中断部署

部署是用户等了很久才触发的操作，如果因为一张图片上传失败就整个中断，体验很差。我们选择记录警告日志但继续执行。用户可以在部署日志中看到哪些文件失败了，下次部署会自动重试。

为什么不在编辑器里实时上传

这是有意的阶段性设计。编辑器内粘贴图片时实时上传到 CDN 是更好的体验，但需要修改编辑器的图片处理逻辑、处理上传进度显示、处理网络异常回退等。我们先把「部署时批量上传」做稳定，编辑器内上传作为后续迭代。

保存路径模板的设计已经为这个场景预留了——它可以给每个文件生成唯一的远程路径，正是编辑器内单图上传需要的。

七、结语

这个功能的核心价值其实就一句话：让部署真正成为一键操作。

之前的 CDN 功能只做了 URL 重写，相当于给了你一把锁但没给钥匙——你还得自己把图片搬到 CDN 仓库。现在补上了「搬运」这一步，整个链路才真正闭环。

从技术角度看，这个功能并不复杂：一个 HTTP 客户端调用 GitHub API，加上 SHA 去重和并发控制。但好的工具设计往往不是炫技，而是把用户不应该操心的事情自动化掉。

写博客应该专注在内容上，而不是和基础设施搏斗。
